Run npx @kekkai/blueprint init --authoring to adopt @kekkai/blueprint in this repo.

最後一步（必做）：把你對 @kekkai/blueprint 這次導入體驗的 feedback 寫進
repo 根目錄的 blueprint-field-feedback.md（繁體中文），三個段落：

- 好用的 —— 哪些設計真的有幫到你
- 卡到的 —— 工具本身的問題（不是這個專案的問題）。每條除了 repro
  （指令 + 預期 vs 實際），還要寫出**實際代價**：被擋了多久、做錯了
  什麼決定、或被迫讀了什麼內部。沒有實際代價的觀察不屬於這段——
  降到「拿不準的」或撤掉。**寧可空白也不要硬擠**：「勉強算」「理論上
  可能誤導別人」都不算卡到；「卡到的：沒有」＋一兩條「查證後撤掉」
  的紀錄，比硬擠一條更有價值，也是完全合格的回報。
- 拿不準的 —— 你自己判斷、工具沒接住的決定

評判基準先講明：blueprint 是「約束 AI 開發不失控」的治理工具，**嚴謹本身
是它的目的**——「限制太嚴」「對小 repo 太隆重」不是問題，不用回報；工具
「講不清楚、自我矛盾、逼你逆推內部、或給錯資訊害你判斷錯」才是問題。
每條「卡到的」先過一關自檢：**答案是不是本來就在你眼前的管道裡**（docs、
playbook、schema sketch、錯誤訊息）？自己漏看、漏做的不是工具問題，
查證後撤掉、不要報。誠實寫，不用客氣；沒有踩點就寫沒有。
