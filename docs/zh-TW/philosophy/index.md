# 運作契約

本章節並非延伸閱讀 —— **它所記載的正是 blueprint 預設藍圖內建的資料本體**。`vuePreset()` / `reactPreset()` 以資料形式承載本章節的每一頁；執行 `init` 後，這些內容即編譯進專案：成為 `docs/architecture-handbook.md`、AI 代理契約，其中機器可檢驗的部分再落為程式碼檢查關卡。修改 Blueprint 組態中的 `principles` / `componentShape` / `playbook`，即等同於修改專案內這份內容的版本。

手冊闡述前端**元件與架構**的設計思維：分層架構、元件形狀、資料邊界、重構紀律。**同一組原則在 Vue 與 React 均成立** —— 兩者的結構對應一致，差異僅在響應式原語（`ref`/`computed` 對應 `useState`/`useMemo`）與生命週期 API。

本手冊的定位是**運作契約，而非討論文件**。凡以 AI 代理協作的專案，建議於第一天即行導入 —— 自始將規則納入管控，專案便能在可控範圍內成長為預期的樣貌，而非等到接近失控時才回頭重構。

## 三級落點

手冊中每條規則落於以下三個位置之一：

| 級別 | 落點 | 意義 |
|---|---|---|
| ✅ | **程式碼檢查／組態** | 有對應規則支撐，於持續整合中自動攔截；設定一次即可。 |
| ◐ | **程式碼檢查（初篩）＋ 代理契約** | 檢查工具僅能標出「需要檢視的位置」（設為警告等級），結論交由程式碼審查判定。 |
| ○ | **代理契約** | 檢查工具無法辨識（涉及語意、流程或人為判斷）—— 編寫為行為規則，由代理在每輪工作中自行遵守。 |

此即 Blueprint 的設計骨架：機器可查核者編譯為 ESLint 組態；唯有審查能判定者編譯為手冊與代理契約。**檢查全數通過不等於架構合格** —— 此即信念第七條。

## Blueprint 的承載方式

| 手冊章節 | Blueprint 載體 |
|---|---|
| 分層架構、模組形狀、套件歸屬 | `architecture` → `emitLint` 與 `inspect` |
| 十條核心信念 | `principles` → 手冊與代理契約 |
| 元件形狀（七條軸線） | `componentShape` → 手冊與代理契約 |
| 資料完整性／執行期負載／重構／協作 | `playbook` → 手冊與代理契約 |
| 量化門檻與自訂規則 | `rules` → `emitLint`（內嵌外掛） |
| 持續整合 | `emit.ci` → `emitCi` |
